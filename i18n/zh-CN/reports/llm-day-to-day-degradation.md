# 🔥 Claude Code Ops 4.6 分析。高推理

当 Anthropic 发布像 Opus 4.6 这样的模型时，**模型权重** — 数十亿学习参数 — 被冻结。训练成本极高（数百万美元、数周计算）。没有人会在一夜之间重新训练模型。

但权重只是更大系统中的一层。研究揭示了至少 **7 种不同的机制**，即使在模型权重冻结时也能导致真实或感知的质量变化。

| 问题 | 回答 |
|----------|--------|
| 模型权重发布后会变吗？ | **不会** — 所有提供商确认 |
| 模型行为是否可能逐日不同？ | **是** — 已证实有 ±8-14% 的波动 |
| 是故意"削弱"吗？ | **不** — 无故意降级证据 |
| 基础设施错误是真实的吗？ | **是** — Anthropic 确认了 3 个影响高达 16% 请求的错误 |
| 部分是心理作用吗？ | **是** — 确认偏见和蜜月效应是真实的 |
| 系统提示/后训练会变吗？ | **是** — 各提供商均有记录 |
| 用户应该相信自己的感知吗？ | **部分** — 存在真实原因，但感知会放大它们 |

## 完整推理栈

模型权重是冻结的，但**其上的九层**可以独立影响你体验的内容：

```
┌──────────────────────────────────────────────┐
│  你的会话上下文                               │  ← 会话内退化
│  (累积错误、长对话)                           │
├──────────────────────────────────────────────┤
│  系统提示                                     │  ← 定期更新
│  (安全规则、行为指令)                         │
├──────────────────────────────────────────────┤
│  后训练 (RLHF / 微调)                         │  ← 可能静默更新
│  (指令遵循、安全对齐)                         │
├──────────────────────────────────────────────┤
│  采样参数                                     │  ← 可在服务器端调整
│  (temperature、top-p、top-k)                  │
├──────────────────────────────────────────────┤
│  推测解码                                     │  ← 草稿模型质量变化
│  (草稿模型预测 + 验证)                        │
├──────────────────────────────────────────────┤
│  MoE 路由 / 批次组合                          │  ← 已证实 ±8-14% 波动
│  (每个请求激活哪些专家)                       │
├──────────────────────────────────────────────┤
│  硬件路由                                     │  ← TPU vs GPU vs Trainium
│  (哪个集群服务你的请求)                       │
├──────────────────────────────────────────────┤
│  量化级别                                     │  ← 负载下可能变化
│  (FP16 vs INT8 vs INT4 精度)                 │
├──────────────────────────────────────────────┤
│  编译器和运行时                               │  ← XLA 错误已证实存在
│  (XLA:TPU、CUDA、硬件特定代码)                │
├──────────────────────────────────────────────┤
│  模型权重（冻结）                             │  ← 这些不会变
│  (数十亿学习参数)                             │
└──────────────────────────────────────────────┘
```

关键心智模型：**冻结权重 ≠ 冻结行为**。这就像说"相同引擎 = 相同驾驶体验"却忽略了轮胎、路况、燃油质量和驾驶员疲劳。

## 已证实原因：基础设施错误

### Anthropic 2025 年 9 月事后分析

2025 年 9 月，Anthropic 发布了详细的事后分析，揭示了 **三个独立的基础设施错误**，导致 2025 年 8 月至 9 月间 Claude 质量下降。他们的官方声明：

> "我们从未因需求、时间段或服务器负载而降低模型质量。用户报告的问题仅由基础设施错误引起。"

### 错误 #1 — 上下文窗口路由错误

Sonnet 4 请求意外被路由到配置为 1M Token 上下文窗口的服务器，而非标准服务器。

- **时间线**：8 月 5 日引入，8 月 29 日负载均衡变更后恶化
- **峰值影响**：最严重的一小时内有 16% 的 Sonnet 4 请求受影响（8 月 31 日）
- **用户影响**：约 30% 的 Claude Code 用户至少有一条消息质量下降
- **隐蔽细节**：路由是"粘性"的 — 一旦你命中坏服务器，后续请求继续发往那里
- **修复**：9 月 4-18 日（跨平台推出）

### 错误 #2 — TPU 输出损坏

TPU 服务器上的配置错误导致 Token 生成期间的错误，为不应该出现的 Token 分配高概率。

- **症状**：英文回复中出现泰语或中文字符，明显的代码语法错误
- **影响**：Opus 4.1 和 Opus 4（8 月 25-28 日），Sonnet 4（8 月 25 日-9 月 2 日）
- **范围**：仅 Claude API；第三方平台不受影响
- **修复**：9 月 2 日回滚

### 错误 #3 — XLA:TPU 编译器错误编译（最棘手）

修复精度问题的代码变更意外暴露了 Google XLA:TPU 中的**潜在编译器错误**。

- **根本原因**：近似 top-k 操作（用于选择最可能的下一个 Token）"有时返回完全错误的结果，但仅针对某些批次大小和模型配置"
- **为什么难以发现**：它的行为根据之前或之后运行的操作以及是否启用调试工具而变化
- **隐藏数月**：2024 年 12 月的先前变通方案意外掩盖了这个更深层的问题
- **影响**：Haiku 3.5 已确认；Sonnet 4 和 Opus 3 的子集疑似受影响
- **解决方案**：从近似 top-k 切换到精确 top-k；接受"轻微效率影响"，因为"模型质量不可妥协"

### 为什么检测困难

Anthropic 自己的自动化评估没有捕捉到用户报告的质量下降，"部分原因是 Claude 往往能从孤立错误中很好地恢复。"每个错误在不同平台上以不同速率产生不同症状，造成"令人困惑的报告混合，不指向任何单一原因。"

关键背景：Claude 运行在**三种不同的硬件平台**上（AWS Trainium、NVIDIA GPU、Google TPU），每种都有不同的故障模式、编译器和精度行为。你的请求可能在不同日期命中不同的硬件。

## 已证实原因：MoE 路由波动

现代大型模型通常使用**混合专家 (MoE)** 架构，其中每个输入仅激活模型参数的一个子集（"专家"）。一个学习的路由器决定使用哪些专家。

Scale AI 的研究揭示了一个关键发现：

> "稀疏 MoE 和批量推理的组合会产生不可预测的结果，因为批次的组成可以决定你的查询被路由到哪个专家，而同一批次中来自其他用户的查询混合是不确定的。"

### 各提供商测量的日间波动

| 提供商 | 日间得分波动 |
|----------|--------------------------|
| OpenAI (GPT-4 变体) | ±10–12% |
| Anthropic (Claude 变体) | ±8–11% |
| Google (Gemini 变体) | ±9–14% |

具体例子：同一个模型一天在越狱抵抗上得分 **77%**，第二天得分 **63%**。相同的模型、相同的权重、相同的测试 — 仅基础设施就造成了 14 个百分点的波动。

这意味着即使零错误和零变更，同一个模型也可以纯粹由于请求如何批处理和路由，在不同日期产生明显不同质量的输出。当日间噪音为 10-15% 时，A/B 测试无法可靠检测 5% 的质量信号。

## 已证实原因：系统提示和后训练更新

### 系统提示变更

模型权重不变，但包装这些权重的**系统提示**可以随时更新。对 Claude 系统提示演进的分析显示了数十次迭代，"热修复" — 添加短指令以修补不良行为 — 定期添加和移除。

Claude 3.7 的系统提示包含多个针对常见 LLM"陷阱"的热修复指令。Claude 4.0 的系统提示移除了所有这些，这些行为通过强化学习在后训练中解决。

### 后训练理论

无法解释的质量变化最合理的理论：公司可以在不更改基础模型权重的情况下更新**微调和 RLHF**（人类反馈强化学习）。这在技术上可以说"模型没有改变"而仍通过更新的安全护栏和指令遵循调整来改变行为。

## 已证实原因：静默模型切换

OpenAI 已被多次记录静默更改用户交互的模型：

- 一夜之间移除模型选择器，强制用户从 GPT-4o 切换到 GPT-5
- 将 GPT-4o 设为隐藏的"遗留模型"，需要在设置中手动切换，应用内无通知
- 一个"自动切换器"错误将用户路由到错误的模型
- Plus 订阅者报告模型在未经同意的情况下切换到"受限版本"

Sam Altman 承认推出"比我们希望的更颠簸一些"。Reddit 帖子获得数千点赞，称新模型是"灾难"和"降级"。

这证明模型切换在行业中**确实发生** — 有时是故意的（产品决策），有时是意外的（路由错误）。

## 贡献因素

### 负载下的量化

为了经济地服务数百万用户，公司可能服务**量化**版本的模型 — 将精度从 FP16 降低到 INT8 或 INT4。这可以将内存使用减少 2-4 倍并加速推理，但会引入细微的质量损失。提供商是否在负载下动态切换量化级别存在争议，但这种技术能力存在并在 vLLM 和 TensorRT 等服务框架中有详细记录。

### 推测解码

现代服务栈使用较小的"草稿"模型预测多个未来的 Token，然后让真实模型验证。理论上这保持相同的输出分布，但在实践中接受率因领域和上下文而异。开箱即用的草稿模型在某些情况下可能工作良好，但通常在领域特定任务或非常长的上下文中挣扎。

### 上下文窗口污染

在长时间的编码会话中，早期的错误在上下文中累积。模型看到自己的错误并可能延续它们。这是单个会话中"Claude 变笨"的最常见原因 — 不是模型退化，而是上下文污染。

**实用提示**：感觉质量下降时使用 `/compact` 或开始新会话。这是你能做的最可行的事情。

## 斯坦福研究 — 及其复杂之处

2023 年斯坦福和 UC Berkeley (Chen, Zaharia, Zou) 的里程碑式研究 — "ChatGPT 的行为如何随时间变化？" — 常被引用为 LLM 退化的证明。标题发现：

> GPT-4 在"这个数是质数吗？一步步思考"上的准确率从 2023 年 3 月的 **97.6%** 下降到 6 月的 **2.4%**。

### 研究证明了什么

- "相同"LLM 服务的行为**可以在短期内大幅变化**
- 不同能力可能朝相反方向变化（GPT-4 数学变差，GPT-3.5 变好）
- 代码生成质量下降（GPT-4 可执行代码：52% → 10%）
- 该研究创造了术语 **"LLM 漂移"**

### 方法论批评

- 3 月版本使用 **temperature 0.0** 而 6 月版本使用 **temperature 1.0** — 一个增加随机性的基本混淆变量
- 每个任务仅 **500 次查询** — 对于确定性统计声明来说太小
- "数学问题"实际上是是/否问题，模型的猜测模式改变了，而非其数学能力
- 变化可能反映了故意的**后训练安全更新**，而非退化

研究证明了重要的事情 — LLM 行为随时间变化 — 但机制可能是故意更新，而非非故意退化。

## 心理学因素

### 确认偏见

一旦有人发推"今天 Claude 很笨"，你就开始注意到每个错误。在没人抱怨的日子里，你忽略同样的错误。社交媒体放大了这种效应。

### 蜜月效应

用户在新模型上经历最初的蜜月期，然后逐渐发现局限性。模型没变 — 预期调整的速度超过了能力应有的速度。

### 任务难度变化

你的任务逐日变化。处理困难问题的一天感觉像是模型变差了，即使它没有。

### "周末 Claude"神话

尽管许多用户相信星期几的模式，但严格分析发现**没有一致证据**支持星期几的质量模式。一篇题为"周一 AI 更笨"的分析毫无发现。

### LLM 的随机本质

LLM 是概率性的。相同的提示每次可以产生不同的输出。在运气不佳时，你可能连续得到几个糟糕的响应 — 纯粹的随机性，不是退化。

## 底线

用户描述的现象是**真实的但被错误归因**：

- **正确**：他们在某些日子的体验变差了
- **不正确**：模型被故意"削弱"

实际原因是以下组合：

1. **基础设施错误** — Anthropic 事后分析证实（影响高达 16% 的请求）
2. **MoE 路由波动** — Scale AI 测量的 ±8-14% 质量波动，即使零变更
3. **系统提示和后训练更新** — 各提供商均有记录
4. **硬件异构性** — TPU vs GPU vs Trainium，每种都有不同的故障模式
5. **上下文污染** — 长会话降低会话内质量
6. **确认偏见** — 社交媒体放大感知的模式
7. **随机波动** — 相同模型、相同提示、每次不同输出

测量问题很严重：±8-14% 的日间波动意味着你无法从噪音中区分真正的 5% 质量变化。这就是为什么"都是你的错觉"和"他们削弱了它"两个阵营都感到自信 — 信噪比使得仅从个人体验中无法区分。

# 🔥 Codex 5.3 高推理与发现

### 报告范围

本节解释为什么用户可能会经历 Claude 输出质量下降的短期窗口，而 Codex 5.3 在编码任务上感觉稳定或更强。重点不是永久模型质量排名。重点是实际服务条件下的短期生产行为。

报告日期：2026 年 3 月 5 日。

### 观察到的模式

报告的模式是：

1. 模型质量在一段时间内可接受
2. 质量似乎下降几天
3. 质量恢复接近以前的基线

这种形状通常是服务栈或推出模式，而非永久基础模型能力变化。永久能力下降通常不会在没有显式回滚或修复的情况下如此快速恢复。

### 高推理：为什么 Codex 5.3 在糟糕窗口中可能看起来更好

Codex 5.3 可以在另一个提供商的降级期间明显更强，原因有几个可能同时发生的技术原因：

1. 产品目标匹配。Codex 5.3 针对代码生成和代理编码工作流进行了优化，因此即使原始模型强度相等，由于工具编排、仓库推理和代码中心指令调优，也可以产生更好的编码结果。
2. 推理策略差异。提供商独立调整延迟、推理深度和解码默认值。一个提供商更保守的策略在另一天的激进速度优化策略下可能看起来"更聪明"。
3. 服务路径分离。即使两个提供商托管最先进的模型，它们运行不同的路由层、编译器/运行时栈和推出流水线。一个栈中的事件不意味着另一个中相关的降级。
4. 推出和回滚时机。如果一个提供商正在进行推出而另一个稳定，用户可以看到大的临时质量分歧，而模型权重没有潜在长期变化。
5. 会话级污染效应。在长编码聊天中，错误累积可能放大感知的下降。竞争助手可能感觉更好，仅因为失败的会话被重置或因为其工具循环恢复得更快。

### 详细发现

对于像"Claude 感觉很弱大约四天，然后恢复了"这样的报告，最可能的解释是：

1. 提供商端事件、路由问题、解码/运行时错误或推出回归影响了请求子集。
2. 问题持续足够长，以至于在真实工作流中被反复注意到。
3. 问题被修复或回滚。
4. 感知质量快速恢复。

在同一期间，Codex 5.3 可能感觉明显更好，因为它不共享相同的事件路径，且编码任务优化放大了实际结果的差距。

### 此模式的假设排序

| 假设 | 可能性 | 理由 |
|------------|------------|-----------|
| 提供商事件加回滚 | 高 | 最匹配数日下降后快速恢复 |
| 服务配置变更（采样/延迟/推理预算） | 高 | 无需模型重新训练的突然行为变化常见来源 |
| 静默别名或快照移动 | 中高 | 可以在无可见用户操作的情况下改变行为 |
| 仅提示漂移和上下文污染 | 中 | 可以降级会话，但不太可能单独解释广泛的多日报告 |
| 永久基础模型退化 | 低 | 与快速恢复到先前质量不一致 |

### 如何确认或否定此发现

要将此从高置信度推断变为确凿证明，收集跨天相同任务集的请求级遥测：

1. 请求时的确切模型标识符和快照/别名。
2. 提供商公开的任何后端指纹或发布标记。
3. 解码参数（temperature、top_p、top_k、max tokens）。
4. 延迟、超时和错误率追踪。
5. 固定编码基准提示集上的结构化质量分数。
6. 失败点的会话长度和 Token 上下文深度。

如果质量下降与事件窗口、配置变更或后端指纹变化相关，则事件/配置假说被确认。如果不存在此类变化且降级仅在长会话中，则上下文污染成为主要解释。

### 实用工程指导

为减少生产中的日间波动：

1. 可用时固定模型快照而非使用浮动别名。
2. 存储请求元数据（模型 ID、参数、延迟、错误、响应质量标签）。
3. 运行固定的每日编码任务金丝雀套件并对回退报警。
4. 多次失败后重置或压缩长运行会话。
5. 为事件窗口保留备用提供商/模型路径。
6. 在内部仪表板中区分"模型质量"和"服务可靠性"。

### 最终结论

在短期 Claude 降级窗口中 Codex 5.3 看起来更好在现代 LLM 运营中是一个技术上合理且预期的结果。最强的解释不是永久模型崩溃。最强的解释是一个提供商的临时服务路径降级，结合同一期间另一个提供商的编码特定优化和稳定运营。
