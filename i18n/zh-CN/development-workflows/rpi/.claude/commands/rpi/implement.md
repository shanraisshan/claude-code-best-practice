---
description: 执行带验证门控的分阶段实施
argument-hint: "<feature-slug> [--phase N] [--validate-only]"
---

## 用户输入

```text
$ARGUMENTS
```

你**必须**解析用户输入以提取功能标识符（`rpi/` 中的文件夹名称）。

## 目的

此命令根据规划文档执行功能的分阶段实施。它编排专业 Agent，执行验证门控，并确保整个实施过程中的章程合规性。

**前置条件**：
- 功能文件夹存在于 `rpi/{feature-slug}/`
- 规划已完成（`rpi/{feature-slug}/plan/PLAN.md` 存在）

**输出位置**：`rpi/{feature-slug}/implement/`

**这是 RPI 工作流的第 4 步**（最后一步 - 实际实施）。

## 标志

- `--phase N`：执行特定阶段编号（1-8），如省略则从阶段 1 开始
- `--validate-only`：仅验证当前阶段，不实施
- `--skip-validation`：跳过验证门控并继续（谨慎使用）

## 可用 Agent

所有 Agent 使用 **Opus 模型**以获得最高质量。

### 实施 Agent

| Agent | 类型 | 何时使用 |
|-------|------|----------|
| `senior-software-engineer` | 自定义 | 所有实施任务 |

### 支持 Agent

| Agent | 类型 | 用途 |
|-------|------|------|
| `Explore` | 内置 | 实施前代码探索 |
| `code-reviewer` | 自定义 | 代码审查和质量验证 |
| `constitutional-validator` | 自定义 | 根据项目章程验证 |
| `documentation-analyst-writer` | 内置 | 文档生成 |

### Agent 路由

所有实施任务由 `senior-software-engineer` Agent 处理。

## 阶段 0：加载上下文和规则

**前置条件**：从用户输入解析功能标识符

**流程**：

### 0.1 加载项目章程

1. 检查仓库中是否存在章程或原则文档
2. 如存在，提取：
   - 技术约束（类型安全、测试、组件隔离）
   - 业务原则（质量标准、工作流）
   - 架构边界
3. 存储约束以在实施过程中执行

### 0.2 加载领域特定指南

根据要修改的文件，加载相关的项目指南：
- 检查组件特定的 README 文件
- 检查编码风格指南
- 检查测试需求文档

### 0.3 分析实施范围

1. 阅读 `rpi/{feature-slug}/plan/PLAN.md`
2. 识别所有要修改的文件
3. 将文件映射到实施 Agent

**输出**：
- 章程上下文摘要
- 已加载的领域规则
- 文件到 Agent 的映射
- 阶段执行计划

**验证**：
- [ ] 章程已加载（如存在）
- [ ] 已为受影响文件加载领域规则
- [ ] 所有文件已映射到 Agent
- [ ] 执行计划已理解

## 分阶段实施工作流

### 阶段实施循环

对于 PLAN.md 中的每个阶段：

```
┌─────────────────────────────────────────────────────────────────┐
│ 阶段 N: [阶段名称]                                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 代码发现 (Explore Agent)                                    │
│     └─→ 在修改代码之前理解现有代码                               │
│                                                                  │
│  2. 实施 (senior-software-engineer)                             │
│     └─→ 实施阶段交付物                                          │
│                                                                  │
│  3. 自我验证                                                    │
│     └─→ 工程师根据阶段清单验证                                   │
│                                                                  │
│  4. 代码审查 (code-reviewer Agent)                              │
│     └─→ 安全性、正确性、可维护性                                 │
│                                                                  │
│  5. 用户验证门控                                                │
│     └─→ 停止并请求用户批准                                       │
│         ├─→ 通过：进入下一阶段                                   │
│         ├─→ 有条件通过：记录问题，继续                           │
│         └─→ 失败：修复问题，重新验证                             │
│                                                                  │
│  6. 文档更新                                                    │
│     └─→ 更新 PLAN.md 中的阶段状态                                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 步骤 1：代码发现（每个阶段）

**Agent**：Explore（内置，通过 Task 工具）

**目的**：在做出变更之前将实施建立在代码现实基础上。

**流程**：
1. 通过 Task 工具启动 Explore Agent，使用 `subagent_type="Explore"`
2. 请求分析当前阶段影响的文件
3. 理解现有模式、集成点、约束

**Explore Agent 提示词**：
```
分析代码库以为 [功能名称] 的阶段 N 实施做准备。

本阶段要修改的文件：
[从 PLAN.md 列出文件]

调查并记录：

1. **当前实现**
   - 这些文件目前如何工作？
   - 使用了什么模式？
   - 哪些函数/类会受到影响？

2. **集成点**
   - 哪些其他文件导入或使用这些模块？
   - 哪些 API 或接口会变更？
   - 哪些测试覆盖此代码？

3. **依赖项**
   - 使用了哪些库？
   - 有哪些内部工具可用？
   - 当前代码存在哪些约束？

4. **需遵循的模式**
   - 这些文件使用什么编码风格？
   - 遵循什么命名约定？
   - 存在什么错误处理模式？

5. **风险和注意事项**
   - 如果我们更改此代码可能会破坏什么？
   - 存在哪些边界情况？
   - 有哪些向后兼容性考虑？

提供发现摘要以指导实施。
```

**输出**：供实施 Agent 使用的发现摘要

---

## 步骤 2：实施（每个阶段）

**Agent**：senior-software-engineer

**流程**：
1. 使用 senior-software-engineer Agent
2. 提供步骤 1 的发现上下文
3. 实施该阶段的所有交付物
4. 遵循章程约束和项目规则

**实施 Agent 提示词模板**：
```
作为 [agent-name] Agent，实施 [功能名称] 的阶段 N 交付物。

## 上下文
- 章程约束：[来自阶段 0]
- 领域规则：[来自阶段 0]
- 发现摘要：[来自步骤 1]

## 阶段 N 交付物
[从 PLAN.md 列出]

## 要修改的文件
[从 PLAN.md 列出文件及具体变更]

## 实施要求
1. 遵循发现中识别的现有代码模式
2. 遵守章程约束（类型安全、测试等）
3. 遵循项目特定规则（如适用）
4. 为新功能编写测试
5. 添加适当的日志记录
6. 优雅处理错误

## 质量清单
- [ ] 代码遵循现有模式
- [ ] 适用处存在类型注解
- [ ] 测试已编写并通过
- [ ] 对现有功能无破坏性变更
- [ ] 为可观测性添加了日志记录
- [ ] 错误处理全面

实施所有交付物并报告已完成的工作。
```

## 步骤 3：自我验证

**Agent**：senior-software-engineer（与步骤 2 相同）

**流程**：
1. Agent 根据阶段清单验证实施
2. 运行 lint（使用项目配置的 linter）
3. 运行与变更相关的测试
4. 验证构建成功

**验证命令**（根据你的项目调整）：

```bash
# 运行 linter
[your-linter-command]

# 运行测试
[your-test-command]

# 构建/编译
[your-build-command]
```

**自我验证清单**：
- [ ] 所有交付物已实施
- [ ] Lint 通过
- [ ] 测试通过
- [ ] 构建成功
- [ ] 现有测试无回归
- [ ] 章程约束已遵守
- [ ] 领域规则已遵循

---

## 步骤 4：代码审查

**Agent**：code-reviewer（自定义，自动调用）

**流程**：
1. 调用 code-reviewer Agent 审查变更
2. 关注正确性、安全性、可维护性
3. 在继续之前解决阻塞问题

**代码审查 Agent 提示词**：
```
作为 code-reviewer Agent，审查 [功能名称] 的阶段 N 实施。

## 变更的文件
[列出修改的文件]

## 所做的变更
[实施摘要]

## 审查重点
- 正确性与测试
- 安全性与依赖管理
- 架构边界
- 清晰优于技巧

## 章程约束
[来自阶段 0]

使用标准输出格式提供审查。
```

**审查结论**：
- **批准**：进入用户验证
- **有建议地批准**：记录建议，继续
- **需要修订**：修复问题，重新审查

---

## 步骤 5：用户验证门控

**关键**：此步骤需要用户交互。不要自动继续。

**流程**：
1. 展示阶段交付物清单
2. 显示已实施的内容（变更的文件、添加的功能）
3. 展示 PLAN.md 中的验证标准
4. 显示代码审查结果
5. **停止并等待用户决策**

**验证请求格式**：
```
## 阶段 N 验证请求

### 已完成的交付物
- [x] [交付物 1] - [实施摘要]
- [x] [交付物 2] - [实施摘要]
- ...

### 变更的文件
| 文件 | 变更类型 | 行数 |
|------|----------|------|
| [文件] | [添加/修改] | [±N] |

### 测试
- [x] 单元测试：通过
- [x] 集成测试：通过
- [x] 构建：成功

### 代码审查
- 结论：[批准 / 有建议地批准]
- 问题：[无 / 列表]

### 验证标准（来自 PLAN.md）
- [ ] [标准 1]
- [ ] [标准 2]
- ...

---

**请验证阶段 N：**
- **通过**：阶段完成，进入阶段 N+1
- **有条件通过**：在下方记录问题，谨慎继续
- **失败**：指定继续前需要修复的问题
```

**用户决策**：
- **通过**：进入下一阶段
- **有条件通过**：记录问题，进入下一阶段
- **失败**：修复问题，重新运行步骤 2-5

---

## 步骤 6：文档更新

**流程**：
1. 更新 `rpi/{feature-slug}/plan/PLAN.md` 的阶段状态
2. 更新 `rpi/{feature-slug}/implement/IMPLEMENT.md` 的验证结果
3. 将每个阶段的验证追加到 IMPLEMENT.md

### 阶段状态跟踪

更新 PLAN.md 中的复选框：
```markdown
- [ ] 阶段 N：未开始
- [~] 阶段 N：进行中
- [x] 阶段 N：已验证（通过）
- [!] 阶段 N：有条件通过（有注释）
- [-] 阶段 N：验证失败（需要返工）
```

### IMPLEMENT.md 模板

```markdown
# 实施记录

**功能**：[feature-slug]
**开始日期**：[日期]
**状态**：[进行中 / 已完成]

---

## 阶段 1：[阶段名称]

**日期**：[日期]
**结论**：[通过 / 有条件通过 / 失败]

### 交付物
- [x] [交付物 1]
- [x] [交付物 2]

### 变更的文件
[包含行数的列表]

### 测试结果
[测试输出摘要]

### 代码审查
[审查结论和注释]

### 备注
[任何额外的备注]

---

## 阶段 2：[阶段名称]
[与阶段 1 相同的结构...]

---

## 摘要

**已完成阶段**：[N] / [N]
**最终状态**：[已完成 / 进行中]
```

## 错误处理

### 实施失败

**如果实施失败**：
1. 记录具体的失败
2. 分析根本原因
3. 尝试替代方法（最多 2 次尝试）
4. 如仍失败，停止并请求用户指导
5. 不要带着不完整的实施进入下一阶段

**消息**："实施失败：[错误]。已尝试 [N] 种方法。需要用户指导。"

### 测试失败

**如果测试失败**：
1. 分析失败原因（代码 bug vs 测试 bug）
2. 修复问题
3. 重新运行测试
4. 如持续失败，记录并请求用户
5. 不要带着失败的测试标记阶段完成

**消息**："测试失败：[失败项]。已尝试修复但未成功。需要用户审查。"

### 构建失败

**如果构建失败**：
1. 检查类型错误
2. 检查缺失的导入
3. 检查语法错误
4. 修复并重新构建
5. 如持续失败，升级给用户

**消息**："构建失败：[错误]。无法自动解决。"

### Agent 失败

**如果 Agent 失败或超时**：
1. 用相同输入重试一次
2. 如仍失败，在没有该 Agent 贡献的情况下继续
3. 在验证请求中记录差距

**消息**："Agent [名称] 失败。在没有其贡献的情况下继续。"

## 注意事项

### 何时使用此命令

- 在 `/rpi:plan` 生成 PLAN.md 之后
- 当需要带验证门控的分阶段实施时
- 对于需要结构化实施的功能

### 何时不使用此命令

- Bug 修复（太重，直接修复即可）
- 非常简单的变更（<30 分钟工作量）
- 探索性原型开发
- 仅文档变更

### 最佳实践

1. **先审查 PLAN.md**：了解你要实施的内容
2. **信任代码发现**：让 Explore Agent 指导实施
3. **遵循现有模式**：让代码发现指导实施
4. **不要跳过验证**：门控存在的目的是尽早发现问题
5. **边做边记录**：每个阶段后更新状态
6. **遇到困难时询问**：宁可询问也不要错误地继续

### RPI 工作流的一部分

第 4 步，共 4 步（描述 → 研究 → 规划 → **实施**）

## 完成后操作

**重要**：完成实施（所有阶段或重大进展）后，始终提示用户压缩对话：

> **上下文管理**：此实施工作流消耗了大量上下文。为保留进度并释放空间，请运行：
>
> ```
> /compact
> ```
>
> 这将总结对话并保留实施状态，同时减少后续工作的 token 使用。

**何时提示压缩**：
- 所有阶段完成后
- 完成每个主要阶段后（如果是多会话实施）
- 如果实施过程中上下文不足
