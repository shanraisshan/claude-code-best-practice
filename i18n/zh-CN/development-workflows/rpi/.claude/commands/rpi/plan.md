## 用户输入

```text
$ARGUMENTS
```

你**必须**解析用户输入以提取功能标识符（`rpi/` 中的文件夹名称）。

## 目的

此命令为功能请求创建全面的规划文档。它在功能的 RPI 文件夹中生成详细规范、技术设计和实施计划。

**前置条件**：
- 功能文件夹存在于 `rpi/{feature-slug}/`
- 研究已完成并有 GO 建议（`rpi/{feature-slug}/research/RESEARCH.md` 存在）

**输出位置**：所有文件保存到 `rpi/{feature-slug}/plan/`

**这是 RPI 工作流的第 3 步**（在 Research 批准 GO 之后）。

## 大纲

1. **加载上下文**：阅读研究报告和项目章程（如存在）
2. **理解需求**：解析功能范围和需求
3. **分析技术需求**：审查架构和依赖
4. **设计架构**：创建高层架构和 API 契约
5. **分解实施**：创建分阶段任务分解
6. **生成文档**：创建结构化文档文件
7. **验证输出**：确保所有质量门控通过
8. **报告完成**：提供摘要和后续步骤

## 阶段

### 阶段 0：加载上下文

**前置条件**：已提供功能标识符

**流程**：
1. **验证研究已完成**：
   - 检查 `rpi/{feature-slug}/research/RESEARCH.md` 是否存在
   - 验证 GO 建议（如为 NO-GO 或 CONDITIONAL 则发出警告）

2. **阅读研究发现**：
   - 提取产品分析
   - 提取技术发现
   - 提取技术可行性评估
   - 记录风险和约束

3. **加载项目章程**（如存在）：
   - 在仓库中查找章程或原则文档
   - 提取相关约束和偏好

**输出**：
- 研究摘要
- 章程上下文（如找到）
- 规划约束

**验证**：
- [ ] 研究报告存在
- [ ] GO 建议已确认
- [ ] 章程已加载（如存在）

---

### 阶段 1：理解功能需求

**前置条件**：阶段 0 完成

**流程**：
1. **从研究报告解析功能描述**：
   - 提取功能名称和主要目标
   - 识别目标组件
   - 理解面向用户 vs 技术功能
   - 确定功能复杂度级别

2. **识别受影响组件**：
   - 主要组件（功能所在位置）
   - 次要组件（集成点）
   - 需要的共享工具
   - 外部依赖

3. **研究现有模式**：
   - 在代码库中搜索类似功能
   - 审查组件架构和模式
   - 识别可重用的代码和模式

**输出**：
- 功能范围文档（内部）
- 受影响组件列表
- 现有模式目录

**验证**：
- [ ] 功能名称和目标已清晰定义
- [ ] 目标组件已识别
- [ ] 功能复杂度已评估

---

### 阶段 2：分析技术需求

**前置条件**：阶段 1 完成

**流程**：
1. **审查组件架构**：
   - 阅读组件 README 和文档
   - 审查现有代码结构
   - 识别使用的架构模式

2. **识别技术依赖**：
   - 内部依赖（其他组件、共享工具）
   - 外部依赖（API、服务、库）
   - 数据库/存储需求
   - 认证/授权需求

3. **评估集成点**：
   - 需要创建或修改的 API
   - 需要的数据库模式变更
   - 事件/消息流
   - 前后端集成

4. **评估技术风险**：
   - 对现有功能的破坏性变更
   - 性能影响
   - 安全问题
   - 数据迁移需求

**输出**：
- 技术需求文档（内部）
- 依赖图
- 集成点图
- 风险评估

**验证**：
- [ ] 组件架构已理解
- [ ] 所有依赖已识别
- [ ] 集成点已映射
- [ ] 技术风险已评估

---

### 阶段 3：设计功能架构

**前置条件**：阶段 1-2 完成

**Agent**：senior-software-engineer

**流程**：
1. **设计高层架构**：
   - 组件/模块结构
   - 数据流图
   - API 接口
   - 数据库模式变更

2. **定义实施方法**：
   - 文件结构和组织
   - 代码组织模式
   - 测试策略
   - 错误处理方法

3. **规划数据库/存储变更**（如适用）：
   - 新集合/表
   - 模式修改
   - 迁移策略
   - 数据验证规则

4. **设计 API 契约**（如适用）：
   - 请求/响应格式
   - 认证要求
   - 错误响应

5. **规划测试策略**：
   - 单元测试需求
   - 集成测试场景
   - 端到端测试用例

**输出**：
- 架构设计文档（内部）
- API 规范
- 数据库模式设计
- 测试策略

**验证**：
- [ ] 高层架构已设计
- [ ] 实施方法已定义
- [ ] 数据库变更已规划（如需要）
- [ ] API 契约已指定（如需要）
- [ ] 测试策略完成

---

### 阶段 4：分解实施任务

**前置条件**：阶段 1-3 完成

**流程**：
1. **识别实施阶段**：
   - 将功能分解为 3-5 个逻辑阶段
   - 每个阶段应交付可工作、可测试的功能
   - 阶段应逐步递进

2. **为每个阶段创建任务分解**：
   - 列出具体实施任务
   - 估算复杂度（低/中/高）
   - 识别任务依赖
   - 分配到适当的代码区域

3. **定义成功标准**：
   - 每个阶段的验收标准
   - 测试需求
   - 文档需求

4. **识别并行化机会**：
   - 可同时进行的任务
   - 前后端并行工作
   - 独立模块开发

**输出**：
- 分阶段实施计划
- 带估算的任务分解
- 每个阶段的成功标准
- 依赖图

**验证**：
- [ ] 功能已分解为 3-5 个逻辑阶段
- [ ] 每个阶段有具体任务
- [ ] 所有任务有复杂度估算
- [ ] 依赖已清晰标记
- [ ] 成功标准已定义

---

### 阶段 5：生成文档

**前置条件**：阶段 1-4 完成

**Agent**：documentation-analyst-writer（通过 Task 工具）

**流程**：
1. **生成 pm.md**（产品需求）：
   - 功能描述和用户故事
   - 章程一致性（如适用）
   - 业务价值和成功指标
   - 用户画像和用例
   - 验收标准
   - 范围外项目

2. **生成 ux.md**（用户体验设计）：
   - 用户界面原型（文本描述）
   - 用户流程和交互
   - 无障碍考虑
   - 错误状态和边界情况

3. **生成 eng.md**（技术规范）：
   - 架构设计
   - API 规范
   - 数据库模式变更
   - 技术栈
   - 技术风险和缓解

4. **生成 PLAN.md**（实施路线图）：
   - 分阶段实施分解
   - 每个阶段的任务列表及估算
   - 依赖和顺序
   - 每个阶段的成功标准
   - 测试需求
   - 验证检查点

**输出文件**（全部保存到 `rpi/{feature-slug}/plan/`）：
- `pm.md` - 产品需求
- `ux.md` - UX 设计
- `eng.md` - 技术规范
- `PLAN.md` - 详细实施路线图

**验证**：
- [ ] 所有 4 个文件存在（pm、ux、eng、PLAN）
- [ ] pm.md 涵盖业务需求
- [ ] ux.md 解决用户体验
- [ ] eng.md 提供技术规范
- [ ] PLAN.md 有分阶段实施
- [ ] 无占位符文本残留
- [ ] Markdown 格式整洁

### Agent 调用

**自定义 Agent**（product-manager、senior-software-engineer、ux-designer）：
- Claude Code 自动从 `.claude/agents/` 检测这些 Agent
- 自然地引用它们："作为 senior-software-engineer Agent..."
- 不需要 Task 工具调用

**内置 Agent**（documentation-analyst-writer）：
- 使用 Task 工具，设置 `subagent_type="documentation-analyst-writer"`

## 错误处理

**如果研究报告不存在**：
- 操作：停止并通知用户
- 消息："未找到研究报告。请先运行 `/rpi:research`。"

**如果研究建议是 NO-GO**：
- 操作：警告用户但允许继续
- 消息："研究建议 NO-GO。是否仍要继续？(y/n)"

**如果目标组件不存在**：
- 操作：与用户确认这是否是新组件
- 消息："未找到组件。这是一个新组件吗？"

**如果文档 Agent 失败**：
- 操作：直接生成文档
- 警告："文档可能不完全符合标准"

---

## 注意事项

- **前置条件**：研究已完成并有 GO 建议
- **RPI 工作流的一部分**：第 3 步，共 4 步（描述 → 研究 → 规划 → 实施）

**最佳实践**：
1. **先审查研究**：确保你理解可行性评估
2. **利用发现**：使用研究阶段的技术发现
3. **具体明确**：详细的计划使实施更顺畅
4. **尽早验证**：在实施前审查文档

**重要**：完成规划工作流后，始终提示用户压缩对话：

> **上下文管理**：此规划工作流消耗了大量上下文。为释放空间用于实施，请运行：
>
> ```
> /compact
> ```
>
> 这将总结对话并保留规划决策，同时减少实施阶段的 token 使用。
